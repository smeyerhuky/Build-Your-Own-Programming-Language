# Extend J0

Use these templates when you want an LLM to help you add a feature to
the J0 lexer or parser. Always include the existing relevant section
of the file so the model has concrete context.

---

## Template: add a new keyword

```
I am extending the J0 compiler (Ch <N>).

Current keyword rules in javalex.l (excerpt):
  "public"    { return(parser.PUBLIC); }
  "class"     { return(parser.CLASS); }
  "while"     { return(parser.WHILE); }
  {id}        { j0.install(yytext()); return(parser.ID); }

I want to add the keyword `<KEYWORD>`.

Rules:
1. Add a `%token <TOKEN_NAME>` to j0gram.y.
2. Add a rule in javalex.l BEFORE the {id} rule.
3. The token constant in parser.java is generated by byacc; do not edit parser.java directly.

Please show:
a. The %token line to add to j0gram.y.
b. The javalex.l rule to add.
c. A minimal grammar rule using this keyword in j0gram.y.
```

## Template: add a new operator / punctuation

```
I am extending J0 to support the operator `<OPERATOR>` (e.g., `!=`, `++`).

Current multi-character operator rules in javalex.l (excerpt):
  "=="  { return(parser.EQ); }
  ">="  { return(parser.GEQ); }

Single-char operators use:
  .     { return(j0.ord(yytext())); }

Is `<OPERATOR>` a multi-character operator? If yes, show:
a. The `%token` line for j0gram.y.
b. The javalex.l rule (remember: BEFORE the catch-all `.` rule).
c. An example grammar rule using it.
```

## Template: diagnose a new keyword being recognised as IDENTIFIER

```
I added the keyword `<KEYWORD>` to javalex.l as:
  "<KEYWORD>" { return(parser.<TOKEN>); }

But my test file outputs token ID for `<KEYWORD>` instead of token <TOKEN>.

My javalex.l keyword section (paste the relevant lines, including the
{id} rule) is:
  <PASTE>

What is wrong? (Hint: check whether the {id} rule appears before my keyword rule.)
```

## Template: explain a shift/reduce conflict

```
byacc produced a shift/reduce conflict warning:
  1 shift/reduce conflict

The conflicting grammar rules from y.output are:
  <PASTE RELEVANT LINES FROM y.output>

Is this conflict safe to ignore for J0, or does it change the meaning
of any valid J0 program? If it changes behaviour, how do I resolve it?
```

---

## Tips for extend-J0 prompts

- Always paste the **existing** rules around where you are inserting,
  not just the new lines.
- Paste the token numbers from your current `parser.java` so the model
  knows which numbers are already taken.
- Ask for `byacc -v` output if conflicts appear — `y.output` has the
  detailed state machine that makes conflicts diagnosable.
